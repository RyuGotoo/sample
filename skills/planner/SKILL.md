# Planner Skill

ユーザー指示を解釈し、ドキュメント生成に必要な項目を計画する。

## 入力

| パラメータ | 説明 | デフォルト |
|-----------|------|-----------|
| ユーザー指示 | ドキュメント生成の要求 | 必須 |
| --template | テンプレートファイルのパス | なし（動的生成） |

## 出力

`workspace/doc-plan.md` - ドキュメント計画

---

## 実行手順

以下の手順を**順番に実行**してください。

### Step 1: パラメータ確認

入力パラメータを確認：
- ユーザー指示: 必須
- テンプレート: オプション

### Step 2: テンプレート有無の判定

| 条件 | アクション |
|------|-----------|
| `--template` 指定あり | テンプレートファイルを読み込み、Step 4 へ進む |
| `--template` 指定なし | Step 3 へ進む |

テンプレートを使用する場合、テンプレートの項目をベースとして使用し、必要に応じて調整する。

### Step 3: 情報充足度チェック

ドキュメント項目を生成するために、以下の5項目すべてが判断可能か確認する：

| 項目 | 充足条件 | 質問例 |
|------|---------|--------|
| 対象読者 | 開発者/運用者/マネージャー/新規参画者 のいずれか判断可能 | 「読者は誰ですか？」 |
| 記述レベル | システム俯瞰/モジュール構造/実装詳細 のいずれか判断可能 | 「どのレベルで書きますか？」 |
| 対象範囲 | 全体/特定モジュール・機能 が明確 | 「対象は全体ですか？特定機能ですか？」 |
| 目的 | オンボーディング/保守/レビュー等 が判断可能 | 「目的は何ですか？」 |
| 出力形式 | 具体的な出力イメージが明確 | 「どのような形式で出力しますか？」 |

**判定基準**:
- 5項目すべて判断可能 → Step 4 へ進む
- いずれかが不明確 → **質問を返して終了**

**質問すべき状況**:
- 「アーキテクチャ概要」「ドキュメント」など抽象的な指示
- 複数の解釈が可能な場合
- 対象読者やレベルが不明確な場合
- 入力ドキュメントの扱い方が不明確な場合（修正対象か参照元か）

**質問例**:
```
ドキュメントを作成するために、以下を教えてください：

1. 対象読者: 開発者 / 運用者 / マネージャー / 新規参画者
2. 記述レベル: システム全体の俯瞰 / モジュール間の連携 / 実装の詳細
3. 対象範囲: コードベース全体 / 特定モジュール（どれ？）
4. 目的: オンボーディング / 設計レビュー / 保守作業の参照
5. 出力形式: どのような形式で出力しますか？
   - 入力ドキュメント: そのまま保持して追加情報を付ける / 参考にして新規作成 / 書き換える
   - 出力イメージ: 例「元記事に参照番号を挿入し末尾に索引追加」「機能ごとにセクション分けした新規ドキュメント」
```

質問を返した場合、ユーザーが回答後に再実行する。

### Step 4: ドキュメント項目の生成

5つの情報に基づいて、ドキュメント項目を生成する。

**ドメイン定義の指針**:
- コードベース全体 / 特定ディレクトリ / 機能領域 / 技術レイヤー
- 明確な境界を持つこと
- Gathererが検索可能な範囲であること

**抽象レベル設定の指針**:
- 「何を含め、何を除外するか」を明確に指示
- 2-3個の具体的な記述例を提供

**対象読者と記述レベルの対応**:

| 対象読者 | 推奨記述レベル | abstractionLevel の例 |
|---------|---------------|----------------------|
| マネージャー | システム俯瞰 | システム全体の構成と主要機能を概観。技術詳細は省略。 |
| 新規参画者 | システム俯瞰〜モジュール構造 | 主要コンポーネントとその役割、基本的なデータフローを説明。 |
| 開発者 | モジュール構造〜実装詳細 | モジュール間の連携、主要API、設計判断の背景を説明。 |
| 運用者 | モジュール構造 | 運用に関連する設定、監視ポイント、障害対応に必要な情報。 |

### Step 5: バリデーション

生成した項目を検証する：

| チェック項目 | 基準 |
|-------------|------|
| ドメインの境界 | 明確で重複がないこと |
| abstractionLevel | 対象読者と目的に適切であること |
| examples | 具体的で参考になること |
| 項目数 | 目的に対して適切な粒度であること |

問題がある場合は修正してから次のステップへ。

### Step 6: 計画出力

`workspace/doc-plan.md` に以下の形式で出力：

```markdown
# ドキュメント計画

## メタ情報

| 項目 | 値 |
|------|-----|
| 対象読者 | {対象読者} |
| 記述レベル | {記述レベル} |
| 対象範囲 | {対象範囲} |
| 目的 | {目的} |
| 出力形式 | {具体的な出力形式の説明} |
| 入力ドキュメント | {入力ドキュメントの扱い方: 保持して追加/参考にして新規作成/書き換え} |
| 対象パス | {target-path} |

## セクション一覧

### sec-1: {タイトル}

- **domain**: {Gathererに渡す検索ドメイン}
- **abstractionLevel**: {何を含め、何を除外するかの明確な指示}
- **examples**:
  - {記述例1}
  - {記述例2}

### sec-2: {タイトル}

...
```

### Step 7: 完了確認

`workspace/doc-plan.md` が生成されていることを確認し、完了を報告。

```
計画完了:
- 出力: workspace/doc-plan.md
- セクション数: N
```

## テンプレート形式

`--template` で指定するテンプレートファイルの形式（Markdown）：

```markdown
# ドキュメント計画テンプレート

## メタ情報

| 項目 | 値 |
|------|-----|
| 対象読者 | 新規参画者 |
| 記述レベル | システム俯瞰〜モジュール構造 |
| 対象範囲 | コードベース全体 |
| 目的 | オンボーディング |
| 出力形式 | 新規ドキュメント作成 |
| 入力ドキュメント | 参考にして新規作成 |

## セクション一覧

### sec-1: システム概要

- **domain**: コードベース全体
- **abstractionLevel**: システム全体の構成と主要コンポーネント間の関係を概観
- **examples**:
  - このシステムは3層アーキテクチャを採用している
  - フロントエンドはReact、バックエンドはNode.jsで構成される

### sec-2: 認証機能

- **domain**: 認証・認可
- **abstractionLevel**: 認証フローと主要なセキュリティ機構を説明
- **examples**:
  - JWTを使用したステートレス認証を採用
  - リフレッシュトークンは7日間有効
```
