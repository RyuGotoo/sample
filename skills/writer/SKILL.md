# Writer Skill

収集結果を対象読者・目的に照らして精査し、最終ドキュメントを生成する。

## 前提条件

- `workspace/doc-plan.md` が存在すること
- `workspace/gatherer-results/{item-id}/extraction-result.json` が存在すること

## 入力

| パラメータ | 説明 | デフォルト |
|-----------|------|-----------|
| --output | 出力ファイル名 | document.md |

## 出力

`workspace/{output}` - 最終ドキュメント

---

## 実行手順

以下の手順を**順番に実行**してください。

### Step 1: 計画の読み込み

`workspace/doc-plan.md` を読み込み、以下を抽出：

- **メタ情報**:
  - 対象読者
  - 記述レベル
  - 対象範囲
  - 目的

- **セクション一覧**:
  - id, title, domain, abstractionLevel, examples

### Step 2: 収集結果の読み込み

各セクションの `workspace/gatherer-results/{item-id}/extraction-result.json` を読み込む。

### Step 3: Precision判断（精査）

各セクションの収集結果を**対象読者・目的**に照らして精査する。

**精査基準**:

| 観点 | チェック内容 | 除外対象 |
|------|-------------|---------|
| 抽象レベル適合 | 記述レベルに合っているか | 詳細すぎる/概要すぎる情報 |
| 対象読者適合 | 読者が理解できる/必要とする情報か | 読者に不要な技術詳細/前提知識 |
| 目的適合 | 目的達成に寄与するか | 目的に無関係な情報 |
| 重複排除 | 他セクションと重複していないか | 重複情報 |

**対象読者別の精査ガイドライン**:

| 対象読者 | 含めるべき情報 | 除外すべき情報 |
|---------|---------------|---------------|
| マネージャー | ビジネス価値、全体構成、主要機能 | 実装詳細、コード例、技術的設定 |
| 新規参画者 | 概念説明、基本フロー、用語解説 | 高度な最適化、エッジケース |
| 開発者 | API仕様、設計判断、コード例 | ビジネス背景の詳細 |
| 運用者 | 設定方法、監視項目、障害対応 | 内部実装の詳細 |

**精査結果**:
各セクションについて、使用する情報と除外する情報を決定。

### Step 4: 文書構造の設計

計画のセクション一覧に基づいて、文書構造を設計：

```markdown
# {ドキュメントタイトル}

## 概要
{目的と対象読者に基づいた導入文}

## {sec-1: タイトル}
...

## {sec-2: タイトル}
...
```

### Step 5: 各セクションの執筆

精査後の情報を使って、各セクションを執筆する。

**執筆ガイドライン**:

| 項目 | 指針 |
|------|------|
| トーン | 対象読者に合わせる（技術的/平易） |
| 詳細度 | 記述レベルに合わせる |
| 例示 | planのexamplesを参考に具体例を含める |
| 構造 | 見出し、箇条書き、表を適切に使用 |

**セクション執筆テンプレート**:

```markdown
## {タイトル}

{セクションの導入文: 何を説明するか}

### {サブセクション1}

{抽出結果からの情報を対象読者向けに書き直し}

### {サブセクション2}

...
```

### Step 6: 文書出力

完成した文書を `workspace/{output}` に出力。

### Step 7: 完了確認

出力ファイルが生成されていることを確認し、完了を報告：

```
執筆完了:
- 出力: workspace/{output}
- セクション数: N
- 精査サマリー:
  - sec-1: 使用 X件 / 除外 Y件
  - sec-2: 使用 X件 / 除外 Y件
  ...
```

## 精査の詳細例

### 例1: 新規参画者向けの場合

**収集結果**:
```
- JWTトークンの署名アルゴリズムはRS256を使用
- トークンのペイロードにはuser_id, role, expを含む
- リフレッシュトークンはRedisに保存、TTL 7日
- 認証ミドルウェアはExpressのbefore hookで実行
```

**精査後（使用）**:
```
- 認証にはJWTトークンを使用
- トークンには利用者情報と有効期限が含まれる
- ログイン状態は7日間維持される
```

**精査後（除外）**:
```
- 署名アルゴリズムRS256 → 新規参画者には不要
- Redisの詳細 → 運用者向け情報
- before hookの実装詳細 → 開発者向け情報
```

### 例2: 開発者向けの場合

同じ収集結果でも、すべて使用し、さらにコード例を追加。

## エラーハンドリング

| 状況 | 対応 |
|------|------|
| doc-plan.mdが存在しない | エラーを報告して終了 |
| extraction-result.jsonが存在しない | 該当セクションをスキップ、警告を出力 |
| 収集結果が空 | 「情報収集中」等のプレースホルダーを出力 |
